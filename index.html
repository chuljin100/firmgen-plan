01

요약

FirmGen은 자연어로 요구사항을 넣으면 STM32 펌웨어 프로젝트를 자동 개발해 주는 플랫폼이다. 코드 조각 생성기가 아니다 — 요구사항 명세부터 핀 설계, 프로젝트 생성, 동작 검증까지를 시스템이 맡는다. 웹으로 품질을 증명하고, 실제 매출은 온프레미스 구축 + 연간 구독에서 만든다.

  • 요구사항 → 자동 개발 — "이 센서로 이걸 하고 싶다"를 넣으면 핀 설계·프로젝트 구조·코드·테스트까지 생성. 수정도 명세 수정으로.
  • FW 인력 리스크 보완 — FW 비전공인 HW 엔지니어도 펌웨어를 만들 수 있게. 1인 개발자 퇴사로 프로젝트가 드랍되는 문제를 지식 자산화로 해소.
  • 웹 (미끼이자 쇼케이스) — 회원가입 시 무료 생성 크레딧 제공. 상용 API(Claude) 기반이라 품질 최상.
  • 온프레미스 (본 수익원) — 오픈 모델 + 임베디드 특화 RAG를 고객 망 내에 구축. 소스코드 외부 유출 0.
  • 가격 구조 — 하드웨어 스펙(=모델 크기=품질)에 따라 3개 티어. 데모와의 품질 격차를 숨기지 않고 업셀 논리로 쓴다.
02

배경과 기회

Copilot·Claude 등 코드 생성 AI는 웹/앱 개발에선 표준이 됐지만, 임베디드 펌웨어는 사각지대다.

  • 보안 규정이 클라우드 AI를 막는다 — 가전·산업장비·자동차 부품 제조사는 펌웨어 소스의 외부 전송 자체가 금지인 경우가 많다. 쓰고 싶어도 못 쓴다. → 온프레미스가 유일한 답.
  • 범용 LLM은 임베디드에 약하다 — HAL API 버전 혼동, 존재하지 않는 레지스터, 핀맵 오류. 데이터시트·레퍼런스 매뉴얼·에라타를 근거로 대는 RAG 없이는 실무 투입이 어렵다.
  • 임베디드 개발자 부족 — 국내 제조사 공통 문제. 시니어 1명 지식을 팀 전체가 쓰게 만드는 도구에 지불 의사가 있다.
  • FW는 1인 개발이 흔하다 — 그 1명이 퇴사하면 프로젝트가 드랍되거나 수정 불능이 된다. 지식이 사람에게만 있고 시스템에는 없다.
  • FW가 전체 일정의 병목이다 — 초기 설계부터 HW·FW가 핀 할당을 협의해야 하고("이 핀은 꼭 이 용도로"), HW가 먼저 끝나도 FW가 안 나오면 제품 전체가 멈춘다.

차별화 핵심: 모델이 아니라 임베디드 특화 RAG 파이프라인이 자산이다. HAL 문서·CubeH5 예제·데이터시트·보드 핀맵 인덱스, 그리고 생성 코드를 arm-gcc로 실제 컴파일해 통과시키는 검증 루프. 모델은 갈아끼워도 이 파이프라인은 남는다.

04

기술 아키텍처

하이브리드 모델 전략

  • 웹 — Claude 등 상용 API. 인프라 투자 없이 최고 품질로 쇼케이스.
  • 온프레미스 — Qwen3-Coder / DeepSeek 계열 오픈 모델 + RAG. 하드웨어 티어가 곧 모델 크기.
  • 품질 격차의 처리 — "상위 티어일수록 데모 품질에 근접"으로 정면 표기. 격차를 숨기면 클레임, 가격표로 만들면 업셀.

요구사항 → 프로젝트 자동 개발 파이프라인

  • ① 대화형 요구사항 명세 — "온도센서 2개 읽어서 임계치 넘으면 릴레이 차단"처럼 쓰면 시스템이 되물어 명세를 완성. 명세가 프로젝트의 원본이 된다.
  • ② 핀·주변장치 설계 — 핀 어시스턴트(아래)가 MCU 핀 할당·주변장치 구성 확정. CubeMX 호환 설정(.ioc) 출력.
  • ③ 프로젝트 생성 — 스니펫이 아니라 빌드 가능한 프로젝트 전체(드라이버·태스크 구조·빌드 스크립트) 생성.
  • ④ 다층 검증 — 컴파일은 최소선일 뿐. 컴파일 게이트 → 정적분석(MISRA-C) → 자동 생성 유닛테스트 → Renode 시뮬레이션 동작 검증 → (상위 티어) 실보드 HIL. "동작이 검증된 펌웨어만 내보낸다"가 제품 보증.
  • ⑤ 수정도 명세로 — 코드 직접 수정 대신 명세 변경 → 재생성. 변경 이력이 명세 단위로 남는다.

핀 어시스턴트 — HW·FW 협업 자동화

초기 설계에서 HW·FW가 하던 핀 협의를 시스템이 대신한다. 데이터시트의 핀 대체기능(AF) 표를 인덱싱해 두고:

  • HW 엔지니어가 회로 구상·고정 핀 제약("이 핀은 꼭 릴레이 구동에")을 입력 → 충돌 검사 → 핀 할당표 확정.
  • 핀맵이 확정되는 순간 FW 뼈대가 즉시 생성 → HW가 끝났는데 FW를 기다리는 직렬 병목 제거.
  • FW 비전공 HW 엔지니어가 이 흐름만으로 동작 펌웨어까지 도달하는 것이 목표 페르소나.

지식 자산화 — 1인 개발 리스크 해소

  • 프로젝트 = 코드가 아니라 명세 + 설계 결정 + 핀맵 + 테스트의 저장소. 담당자가 퇴사해도 명세에서 재생성·수정 가능.
  • "퇴사 보험"은 그 자체로 영업 포인트이자 연간 구독 유지 논리.

공통 기반 (웹·온프렘 동일)

  • RAG 인덱스 — STM32 HAL 드라이버 문서, CubeH5 예제 코드, 레퍼런스 매뉴얼(RM0481 등), 에라타, 데이터시트 핀 AF 표, Nucleo/Discovery 보드 핀맵.
  • 프로젝트 컨텍스트 — 온프렘은 고객 사내 코드베이스를 추가 인덱싱(선택) → 사내 코딩 컨벤션·기존 드라이버를 반영한 생성. 클라우드가 절대 못 주는 가치.

칩 지원 확대 = RAG 인덱스 추가 작업. 모델 재학습 불필요 → NXP·ESP32·Nordic으로 수평 확장 용이.

06

로드맵

  • 1단계 · M1–M3
    웹 MVP

    회원가입·크레딧·프롬프트→코드 생성(Claude API + STM32 RAG). 구축 문의 폼. 이 단계에서 바로 리드 수집 시작.

  • 2단계 · M4–M6
    검증 파이프라인 + 오픈 모델 PoC

    컴파일 게이트 + 정적분석 + 자동 유닛테스트 + Renode 시뮬레이션. 핀 어시스턴트 프로토타입. Qwen3-Coder-30B를 DGX Spark에서 벤치마크 → Lite 티어 사양·가격 확정.

  • 3단계 · M7–M9
    온프레미스 패키지 v1

    설치 자동화(오프라인 인스톤러), 관리 콘솔, 라이선스 서버. Lite 티어 판매 개시.

  • 4단계 · M10–M12
    첫 레퍼런스 구축 + 상위 티어

    파일럿 고객 1곳 구축 사례화. Team/Enterprise 티어 오픈. 칩 지원 확대(ESP32 또는 NXP) 착수.
